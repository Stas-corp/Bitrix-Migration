## Подход к миграции файлов Bitrix24 → Odoo

### Проблема

Стандартный xlsx-импорт Odoo не поддерживает загрузку файлов — только структурированные данные (проекты, задачи, комментарии). Файлы нужно загружать отдельным процессом после основного импорта.

### Решение

Python-скрипт, который по SFTP читает файлы с сервера Bitrix и через XML-RPC API создаёт записи `ir.attachment` в Odoo с привязкой к нужным сущностям.

### Порядок действий

**Шаг 1 — Выгрузка маппинга** Три SQL-запрос дают CSV-файл с информацией: какой файл на диске Bitrix к какой сущности относится. Для каждого файла — его имя, mimetype, относительный путь на диске и external_id записи-владельца (например `project_task_12345`).

**Шаг 2 — Резолвинг external_id.** После xlsx-импорта в Odoo каждая запись получает external_id в таблице `ir.model.data`. Скрипт одним вызовом загружает весь маппинг `external_id → odoo_id` для нужной модели. Это позволяет понять, что `project_task_12345` из Bitrix — это запись с `id=789` в Odoo.

**Шаг 3 — Загрузка файлов.** Для каждой строки CSV читает файл с сервера Bitrix через SFTP, кодирует в base64 и создаёт `ir.attachment` через XML-RPC с полями `res_model` и `res_id` — это привязывает файл к конкретной задаче или встрече.

### Упрощение по комментариям

Файлы из комментариев привязываются не к самому комментарию (`mail.message`), а к задаче, в которой был комментарий. Файл появится в chatter задачи как вложение. Точная привязка к конкретному комментарию потребовала бы сложного маппинга через дату+автора, а выигрыш минимальный — пользователь всё равно увидит файл в контексте задачи.

### Объёмы

Суммарно ~126K файлов: 48K в задачах, 75K в комментариях, 288 во встречах. Каждый файл — отдельный HTTP-запрос к Odoo. При средней скорости 2–5 файлов/сек полная загрузка займёт ~7–17 часов. Скрипт логирует прогресс каждые 100 файлов с оценкой оставшегося времени, а ошибки сохраняет в отдельный CSV для повторной обработки.
