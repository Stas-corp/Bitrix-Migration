Три блока проблем:

1. **Права на `project.task`** — критично. Пользователь создаёт задачу, назначает на другого — и сразу теряет доступ.
2. **Не мигрировали руководители отделов и менеджеры сотрудников** — важно для оргструктуры и прав.

## Диагноз: что именно сломано в правах

### Корень проблемы

В [security/bitrix_task_employee_security.xml](addons/bitrix_migration/security/bitrix_task_employee_security.xml) record rule для `project.task` (group `group_bitrix_task_employee`) пускала пользователя только если он:

- выполнитель (`user_ids`),
- наблюдатель (`x_bitrix_auditor_employee_ids.user_id`),
- менеджер проекта (`project_id.user_id`),
- автор **проекта** (`project_id.create_uid`).

**Отсутствовали критичные условия:**
- `('create_uid', '=', user.id)` — автор самой **задачи**;
- `('message_partner_ids', 'in', user.partner_id.id)` — follower задачи.

### Аналогия с двумя замками

Представь задачу как комнату с двумя замками:

1. **Нативный замок Odoo** — открывается если ты follower. Одоо автоматически делает create_uid follower-ом при создании → первый замок открыт.
2. **Наш кастомный замок** — открывается только если ты assignee / auditor / project manager. Халабаев назначил задачу на другого → он не assignee → **второй замок не открывается**.

Итог: Халабаев имеет ключ от первого замка, но не от второго. Ошибка "no read/write access".

Дополнительно: ошибка **write**, а не только read — даже если пользователь читает задачу как follower, **наше правило блокирует запись**, потому что не содержит проверки `message_partner_ids`.

### Почему `privacy_visibility='followers'` не спасает

`privacy_visibility='followers'` на `project.project` означает:
- Задачи видны тем, кто **follower проекта** ИЛИ **follower конкретной задачи** ИЛИ **assignee**.
- Это не значит "каждый видит свои". "Свои" = задачи, где ты follower/assignee.

Нативное Odoo-правило для `project.task` проверяет `message_partner_ids` — и пустило бы Халабаева. Но **наше кастомное правило** этого не делало и блокировало write.

### Матрица "кто что видит" (до фикса)

| Пользователь | Нативный read | Наше read | Нативный write | Наше write | Итог |
|---|---|---|---|---|---|
| Assignee | ✓ | ✓ | ✓ | ✓ | Работает |
| Creator (не assignee) | ✓ (follower) | ✗ | ✓ | ✗ | **write ошибка** |
| Auditor | ✗ | ✓ | ✗ | ✓ | Зависит от subscribe |
| Project manager | ✓ | ✓ | ✓ | ✓ | Работает |
| Руководитель исполнителя | ✗ | ✗ | ✗ | ✗ | **Ничего не видит** |

## Архитектура `privacy_visibility` и дискуссия

### Вопрос: если поставить `employees` на "Бітрікс: Без проекта" — все увидят все задачи?

**Да.** `privacy_visibility='employees'` открывает все задачи проекта всем internal users.  
Это проблема — в Bitrix "задачи без проекта" часто личные (1-на-1 между двумя людьми).

### Варианты архитектуры (были рассмотрены)

| Вариант | Код | Приватность | Минусы |
|---|---|---|---|
| **A. `project_id = False` (native private tasks)** | ~0 | Native | Нет единого kanban для всех |
| B. Follower-auto-subscribe в fallback-проекте | средне | Через followers | Нужна subscribe-логика + `allowed_user_ids` |
| C. Персональный проект на каждого юзера | много | Native | Сотни пустых проектов |
| D. Две корзины: личные + общие | средне | Смешанная | Нужна логика классификации |

### Итоговая рекомендованная архитектура (три слоя)

1. **Нативный слой Odoo** — `project_id = False` для Bitrix-задач без `GROUP_ID`. Odoo нативно ограничивает видимость до `user_ids` + followers.
2. **Auto-subscribe** — в `project.task.create/write` подписывать creator + R + A + U. Убирает баг "создал и потерял доступ" без custom rule.
3. **`x_bitrix_allowed_user_ids`** (stored computed) — только для иерархии руководителей. Простой record rule: `[('x_bitrix_allowed_user_ids', 'in', user.id)]`.

> Это **будущий архитектурный рефакторинг**. Для немедленного фикса достаточно минимального патча.

## Применённый фикс (выполнено 2026-04-24)

### Файл: `security/bitrix_task_employee_security.xml`

**До:**
```python
['|', '|', '|',
  ('user_ids', 'in', user.id),
  ('x_bitrix_auditor_employee_ids.user_id', '=', user.id),
  ('project_id.user_id', '=', user.id),
  ('project_id.create_uid', '=', user.id)
]
```

**После:**
```python
['|', '|', '|', '|', '|',
  ('create_uid', '=', user.id),
  ('message_partner_ids', 'in', user.partner_id.id),
  ('user_ids', 'in', user.id),
  ('x_bitrix_auditor_employee_ids.user_id', '=', user.id),
  ('project_id.user_id', '=', user.id),
  ('project_id.create_uid', '=', user.id)
]
```

### Применение

```bash
docker exec -e PGPASSWORD=odoo_pass bitrix_migration_odoo \
  odoo -c /etc/odoo/odoo.conf -d odoo -u bitrix_migration --stop-after-init --db_password=odoo_pass
```

