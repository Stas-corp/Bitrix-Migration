По скрінах це **не один баг**, а три різні блоки:

1. **права на задачі `project.task`** — критично;  
2. **не домігрувалися/не заповнилися керівники відділів і менеджери співробітників** — важливо для оргструктури і майбутніх прав;  
3. **неперекладені елементи в картці співробітника** — косметика, але треба закрити до запуску.

## **1\. Помилка доступу при створенні задачі на інших співробітників**

Тут головний скрін — де Odoo пише:

немає read/write access to:  
 Завдання `project.task`

Це означає: користувач **може відкрити форму створення задачі**, але коли він ставить задачу на іншого співробітника, правило доступу перестає його пускати до цієї задачі.

Типова причина в Odoo така: є rule на `project.task`, наприклад “бачити тільки свої задачі / задачі, де ти виконавець / follower”. Коли Халабаєв створює задачу і ставить її на іншого співробітника, він сам уже не є виконавцем. Після збереження Odoo перевіряє доступ до створеного запису — і користувач раптом не має права його читати або писати.

Odoo працює у два шари: спочатку модельні права `ir.model.access`, а потім record rules, які перевіряються по кожному запису окремо. Тобто навіть якщо в користувача є загальне право на модель `project.task`, record rule все одно може заборонити конкретну задачу.

### **Що фіксити першим**

Спочатку перевірити проєкт **“Бітрікс без проекта”** і всі подібні службові проєкти, в яких користувачі мають створювати задачі.

В Odoo в налаштуваннях проєкту є visibility. Якщо проєкт стоїть як **Invited internal users / private**, то доступ до проєкту і його задач мають тільки followers проєкту та користувачі з правами Project Administrator. Якщо стоїть **All internal users**, тоді всі внутрішні користувачі мають доступ до проєкту і всіх його задач.

Тому швидкий фікс на staging:

**Project → “Бітрікс без проекта” → Settings → Visibility → All internal users.**

Або, якщо не хочете відкривати всім внутрішнім користувачам, тоді треба додати всіх потрібних співробітників/керівників у followers/project members цього проєкту. Але для вашого кейсу з міграцією з Бітрікс “Бітрікс без проекта” скоріше має бути службовим загальнодоступним проєктом для внутрішніх користувачів.

## **2\. Якщо задачі не мають бути видимі всім — потрібне кастомне правило**

Якщо бізнес-логіка така:

* звичайний співробітник роздробу може створити задачу;  
* може поставити задачу іншому співробітнику;  
* директор шоуруму бачить задачі своїх людей;  
* регіональний керівник бачить задачі своїх підлеглих/відділів;  
* але всі задачі всім відкривати не можна,

тоді простого налаштування visibility буде мало. Треба робити кастомну security-логіку.

Я б зробив так:

### **Мінімальне правило для задач**

Користувач має мати read/write до задачі, якщо він:

* створив задачу;  
* є виконавцем;  
* є follower;  
* є керівником виконавця;  
* є керівником відділу виконавця;  
* є регіональним керівником над цим відділом;  
* є project manager / project admin.

Особливо важливий пункт: **creator задачі має зберігати read/write доступ до задачі**, навіть якщо поставив її на іншого користувача. Інакше ви якраз отримуєте помилку після створення.

Технічно я б не радив городити дуже складний domain прямо в `ir.rule`, бо HR-ієрархія швидко стає болючою. Краще зробити в кастомному модулі обчислюване поле типу:

`allowed_user_ids`

і записувати туди:

* автора задачі;  
* виконавців;  
* project manager;  
* керівників виконавців;  
* регіональних керівників;  
* інших дозволених користувачів за вашою оргструктурою.

А record rule тоді простий:

“користувач може бачити/редагувати task, якщо він входить в `allowed_user_ids`”.

Це буде значно стабільніше для go-live.

## **3\. Перевірити групи користувачів**

Окремо треба перевірити, що Халабаєв, директор шоуруму і регіональний керівник мають не тільки базовий доступ “Employee”, а й нормальні права на Project/Tasks.

В адмінці:

**Settings → Users → конкретний користувач → Access Rights / Technical Access Rights**

Потрібно перевірити:

* чи є доступ до Project;  
* чи є право створювати задачі;  
* чи не сидить користувач у занадто вузькій кастомній групі;  
* чи не створений кастомний record rule, який дозволяє тільки “мої задачі”.

Odoo дозволяє керувати правами через групи, а record rules задаються доменами, які фільтрують доступ до даних. В документації прямо попереджають, що record rules можуть суттєво змінювати видимість і редагування записів.

На staging я б зробив так:

1. Відкрити **Settings → Technical → Security → Record Rules**.  
2. Відфільтрувати по моделі **Project Task / `project.task`**.  
3. Подивитися всі кастомні правила.  
4. Тимчасово вимкнути підозріле правило на staging.  
5. Повторити сценарій Халабаєва.  
6. Якщо помилка зникла — правило знайдено, його треба переписувати.

Не треба “фіксити” це тим, що всім роздати Project Administrator. Це обійде помилку, але поламає модель доступів.

## **4\. “Немає полів / при натисканні на поля помилка доступу”**

Це, скоріш за все, пов’язано з тим самим блоком прав.

На першому скріні в задачі видно порожню зону/поля, і тестер пише, що при натисканні вилітає access error. Ймовірні причини:

* поле посилається на запис, який користувач не має права читати;  
* many2one/many2many поле відкриває модель, до якої нема read access;  
* у view є поля з обмеженням `groups="..."`, а користувач не входить у потрібну групу;  
* кастомні поля з міграції додані в форму, але доступ до пов’язаної моделі не налаштований;  
* задача або проєкт приватні, і пов’язані записи не читаються.

Що робити:

1. Спочатку виправити доступ до `project.task` і visibility проєкту.  
2. Потім повторити сценарій.  
3. Якщо помилка лишилась — натиснути поле і подивитися точний текст access error: там буде модель, наприклад `project.task`, `project.project`, `hr.employee`, `res.users`, кастомна модель тощо.  
4. Для цієї моделі додати read access або виправити record rule.  
5. У developer mode відкрити view задачі і перевірити, чи немає на полях неправильних `groups`.

Тут важливо: якщо поле не показується через групи — це не баг міграції даних, а view/security. Якщо поле показується, але при кліку access error — це права на модель або конкретний запис.

## **5\. Відсутні керівники відділів**

Це вже окрема проблема міграції HR-структури.

На скріні в картці співробітника видно, що є відділ, є organization chart, але тестер пише: **“в відділах відсутні керівники відділів”**.

В Odoo є різні речі:

| Що | Де в Odoo | Для чого |
| ----- | ----- | ----- |
| Відділ співробітника | `hr.employee.department_id` | де працює людина |
| Прямий керівник співробітника | `hr.employee.parent_id` / Manager | оргструктура співробітника |
| Керівник відділу | `hr.department.manager_id` | хто керує відділом |
| Пов’язаний користувач | `hr.employee.user_id` | щоб співробітник міг бути assignee/user в задачах |

Схоже, у вас частково підтягнулася оргструктура співробітників, але **не заповнилося поле manager на самих відділах**.

Odoo при створенні/редагуванні співробітника має окремі поля Department, Manager і Coach; якщо в Department налаштований manager/coach, вони можуть автоматично підтягуватися в картку співробітника. У списку відділів Odoo також показує окрему колонку Manager, тобто це самостійне поле відділу, а не те саме, що organization chart співробітника.

### **Як фіксити**

Потрібно доробити міграційний скрипт по HR:

1. З Bitrix взяти структуру відділів.  
2. Для кожного відділу знайти керівника.  
3. У Odoo заповнити:  
   * `hr.department.parent_id`;  
   * `hr.department.manager_id`;  
   * `hr.department.company_id`, якщо multi-company;  
   * `hr.employee.department_id`;  
   * `hr.employee.parent_id`;  
   * `hr.employee.user_id`, якщо співробітник є користувачем системи.  
4. Перевірити, що немає циклів типу “А керівник Б, а Б керівник А”.  
5. Перевірити, що керівник відділу сам належить до правильного відділу або до батьківського відділу.

Це важливо не тільки для красивої оргструктури. Якщо ви будете будувати права на задачі через “керівник бачить задачі підлеглих”, без коректного `hr.employee.parent_id` і `hr.department.manager_id` воно буде постійно ламатися.

## **6\. Переклад у картці співробітника**

На скріні видно англійські шматки:

* `WORK`  
* `ORGANIZATION CHART`  
* можливо ще `Zoom`

Це не критичний функціональний баг, але перед запуском краще прибрати.

Фікс:

1. Увімкнути українську мову.  
2. Зайти в **Settings → Translations → Import / Export → Export Translations**.  
3. Експортувати переклади для модулів, які дають цю форму: найімовірніше `hr`, `hr_org_chart` або ваш кастомний модуль/тема.  
4. Додати переклади в `uk.po`, наприклад:  
   * `Work` → `Робота` або `Робоча інформація`;  
   * `Organization Chart` → `Оргструктура`;  
   * `Zoom` → `Масштаб`.  
5. Покласти файл у `your_module/i18n/uk.po`.  
6. Оновити модуль.

Odoo дозволяє експортувати translatable terms через Settings → Translations → Import/Export, а текстові атрибути views, зокрема `string`, `help`, `placeholder` та текстові вузли, потрапляють у переклад.

Швидкий варіант для staging: вручну знайти ці терміни в **Settings → Technical → Translations → Translated Terms** і дописати переклади. Але правильніше для go-live — тримати це в `i18n/uk.po` вашого кастомного модуля, щоб після оновлень усе не загубилося.

## **Що я б поставив у пріоритет**

| Пріоритет | Проблема | Що робити |
| ----- | ----- | ----- |
| P0 | Access error на задачах | Налаштувати visibility проєктів \+ права Project \+ record rules на `project.task` |
| P0 | Неможливо створювати задачі на інших людей | Додати creator/project manager/керівників у дозволених користувачів задачі |
| P1 | Немає керівників відділів | Домігрувати `hr.department.manager_id` і `hr.employee.parent_id` |
| P1 | Керівники не бачать задачі підлеглих | Будувати правила доступу після виправлення HR-ієрархії |
| P2 | Англійські лейбли в картці співробітника | Додати переклади через `uk.po` або Translated Terms |
| P2 | Порожні/недоступні поля в задачі | Перевірити після фіксу прав; якщо лишиться — дивитися конкретну модель в access error |

Мій висновок: **починати треба не з перекладу і не з UI, а з прав на `project.task` і visibility проєкту “Бітрікс без проекта”**. Дуже ймовірно, що після цього частина “порожніх полів / помилка при натисканні” теж зникне. Потім уже домігрувати керівників відділів, бо без цього нормальні правила “керівник бачить задачі своїх людей” не збудуються.
