Это документация по тем шагам которые я сделал для выгрузки данных из bitrix.
Мне нужно решение для миграции их в Odoo, у меня версия Community Edition 19
---

Да, для **Odoo Community 19** у вас есть рабочий путь миграции, и по вашему README это лучше делать не как “один большой импорт”, а как **гибридную миграцию**: основные сущности загружать через стандартный import Odoo, а историю, комментарии и вложения — через кастомный загрузчик внутри Odoo. В вашей выгрузке уже есть хорошая база: **96 проектов/групп, ~135 855 задач, 214 уникальных тегов, 535 стадий, до ~418 000 реальных комментариев и 5 821 встреча**. 

Почему именно так: в Odoo 19 есть стандартный импорт с **External ID**, и связанные записи можно связывать через поля вида `Field/External ID`; связанные сущности нужно импортировать раньше тех, что на них ссылаются. В 19-й версии Odoo также добавили готовые import templates для типовых сущностей, включая задачи. У задач есть **несколько assignees**, есть **sub-tasks**, а стадии задач в Odoo **по умолчанию привязаны к проекту**, хотя могут быть общими для нескольких проектов. ([Odoo][1])

Из этого следует такая правильная модель миграции.

**1. Что переносить стандартным импортом Odoo**

Сначала загрузите:

* пользователей и контакты, чтобы потом было на кого вешать задачи, комментарии и встречи;
* единый справочник тегов;
* проекты;
* стадии задач;
* задачи.

Это хорошо ложится на ваш README: проекты идут из `b_sonet_group`, задачи — из `b_tasks`, теги надо объединять **по имени**, а не по ID, потому что в Bitrix одна и та же метка дублируется множеством записей; стадии Bitrix — это не “стадии проекта”, а именно **канбан-стадии задач**. Поэтому в Odoo их нужно маппить не в “статус проекта”, а в task stages.  ([Odoo][2])

**2. Что переносить кастомным модулем/скриптом**

Через серверный импортёр в Odoo нужно грузить:

* комментарии задач;
* вложения задач;
* вложения комментариев;
* встречи;
* при необходимости комментарии встреч.

Причина простая: стандартный import Odoo официально покрывает импорт данных и relation-полей, а загрузка файлов в этом потоке документирована именно для **Image columns**. То есть для исторических chatter-комментариев и обычных вложений к задачам/сообщениям стандартный UI-import — плохой инструмент. ([Odoo][1])

**3. Как именно я бы замапил ваши данные в Odoo CE 19**

Проекты и рабочие группы из Bitrix я бы в обоих случаях заводил как `project.project`, но добавил бы кастомные поля:

* `x_bitrix_id`
* `x_bitrix_entity_type` = project / workgroup
* `x_bitrix_closed`
* `x_bitrix_owner_id`

Это нужно потому, что в вашем источнике `b_sonet_group` хранит и проекты, и рабочие группы, а реального “этапа проекта” там нет — только признак `CLOSED = Y/N`. Значит, не надо выдумывать project stage в Odoo; лучше хранить исходный тип и признак закрытия отдельно. `partner_id` пока оставлять пустым, пока не появится отдельная связка с CRM/контрагентами. 

Стадии я бы переносил **не дедуплицируя глобально до 44 названий**, а сохраняя их **по проектам**, потому что у вас в Bitrix стадии привязаны к конкретным группам, и Odoo как раз поддерживает stages, специфичные для проекта. Из вашего README это очень хорошо совпадает с моделью Odoo. Мой совет: в первой волне брать только стадии `ENTITY_TYPE='G'`, а `A`-стадии потоков не тащить, пока не подтвердите, что они реально используются задачами в работе.  ([Odoo][2])

Задачи из `b_tasks` маппятся хорошо:

* `TITLE` → название;
* `GROUP_ID` → проект;
* `responsible_user_ids` → assignees;
* `tags` → task tags;
* `DEADLINE` → deadline;
* `DESCRIPTION` → description;
* `PARENT_ID` → parent task.

Так как Odoo поддерживает **несколько assignees** и **sub-tasks**, ваш набор полей для задач подходит. Но подзадачи нужно связывать **во второй проход**: сначала создать все задачи с External ID, потом обновить поле родителя через `Parent Task/External ID`.  ([Odoo][3])

**4. Что я бы не переносил 1:1**

Я бы **не тянул в chatter Odoo**:

* системные сообщения;
* автоуведомления;
* технические логи изменения статусов.

В вашем README уже выделено, что в `b_forum_message` есть реальные комментарии, автосообщения и системные логи. Для Odoo полезна только человеческая коммуникация; системные логи превратят chatter в мусор. Поэтому в Odoo стоит переносить только блок `SERVICE_TYPE IS NULL AND NEW_TOPIC = 'N'`, а всё остальное — максимум в отдельный архивный JSON/CSV. 

**5. Комментарии и вложения**

Для комментариев нужен отдельный импорт в chatter-модель Odoo с сохранением:

* связи с задачей;
* автора;
* исходной даты;
* bitrix_message_id для дедупликации;
* HTML/текста комментария.

Критичный момент: в Odoo автором chatter-сообщения обычно является **partner**, а не просто user ID. Значит, без отдельной таблицы соответствия `Bitrix user ID → Odoo user → Odoo partner` вы не восстановите ни комментарии, ни встречи корректно. По вашему README это видно сразу: `user_id`, `responsible_user_ids`, `AUTHOR_ID`, `MEETING_HOST`, `participants` везде опираются на Bitrix user IDs. 

По вложениям решение такое:

* файлы задач цеплять к задаче;
* файлы комментариев цеплять либо к импортированному сообщению, либо к задаче с пометкой `x_bitrix_forum_message_id`;
* физические файлы брать не из БД, а из снапшота `upload/`, потому что в README у вас есть только относительный `file_path`, ведущий в файловую систему Bitrix. 

**6. Встречи**

Встречи из `b_calendar_event` можно переносить в `calendar.event`, но я бы делал это **отдельной фазой**, не смешивая с задачами. У вас уже правильно дедуплицированы встречи по `ID = PARENT_ID`, а участники собираются из дочерних записей. Это хороший источник. Но снова нужен user mapping. Если календарь в Bitrix был второстепенным инструментом, его можно отложить на post-go-live и сначала мигрировать только проекты/задачи/историю по задачам. 

**7. Что я рекомендую как итоговое решение под Community 19**

Самый здравый вариант для вас:

1. Делаете кастомный модуль `bitrix_migration`.
2. В него добавляете на ключевые модели поля `x_bitrix_id`, `x_bitrix_stage_id`, `x_bitrix_message_id`, `x_bitrix_file_path`.
3. Проекты, теги, стадии и задачи загружаете через стандартный import Odoo с External ID.
4. Комментарии, вложения и встречи грузите серверным Python-скриптом через ORM Odoo пакетами.
5. Все связи строите не по database id, а по `External ID`/`x_bitrix_id`.

Это решение **не требует Enterprise**, не опирается на Studio и нормально подходит под CE 19. По сути, standard import закрывает “каркас”, а custom loader переносит “историю”.

**8. Порядок миграции**

Практически я бы запускал в таком порядке:

* пользователи / контакты;
* единый справочник тегов;
* проекты;
* project task stages;
* задачи без parent;
* второй проход по parent/subtask;
* реальные комментарии;
* файлы задач;
* файлы комментариев;
* встречи;
* комментарии встреч.

Так вы избежите развала ссылок и сможете делать повторный импорт без дублей за счёт External ID и `x_bitrix_*` ключей. Odoo прямо рекомендует использовать External ID для импорта из сторонних систем и для последующих обновлений тех же записей. ([Odoo][1])

Мой вывод: **миграция реальна и вполне контролируема**, но для вашего объёма правильная граница такая — **UI import для master data, кастомный ETL для history data**. Если делать всё через CSV/UI, вы упрётесь именно в comments + attachments + idempotency. Если делать всё только кодом, вы потратите лишнее время там, где Odoo уже умеет импортировать из коробки.

Опорный документ, на который я опирался: [README по выгрузке Bitrix → Odoo](sandbox:/mnt/data/README.md)

## Готовое ТЗ для Claude

**Роль:** ты senior Odoo developer и migration architect.
**Задача:** спроектировать и реализовать безопасную, идемпотентную миграцию из Bitrix24 в Odoo 19 Community Edition через **внутренний Python/ORM-способ**, без ставки на XML-RPC/старый RPC, с учётом архитектуры Bitrix и модели данных Odoo.

### 1) Цель миграции

Нужно перенести из Bitrix в Odoo:

1. проекты и рабочие группы;
2. задачи и подзадачи;
3. теги;
4. канбан-стадии задач;
5. реальные комментарии задач;
6. встречи календаря;
7. вложения задач, комментариев и встреч;
8. при необходимости — комментарии встреч.

Результат должен быть таким, чтобы Odoo вел себя **нативно по-oдушному**, а не как “копия Bitrix внутри Odoo”. Это означает: в Odoo должны использоваться собственные модели `project.project`, `project.task`, `project.task.type`, chatter и attachments; Bitrix-специфичные особенности нужно сохранять в кастомных `x_bitrix_*` полях и технических таблицах соответствия, а не ломать стандартную модель Odoo. Архитектурно это лучший путь, потому что Odoo 19 уже поддерживает проектные стадии задач, подзадачи и chatter-механику из коробки. ([Odoo][2])

### 2) Базовые архитектурные решения

Используй **внутренний ORM-мигратор в кастомном модуле** `bitrix_migration`, запускаемый через `odoo-bin shell` или через сервисную модель-раннер. Не строй решение вокруг прямой записи в PostgreSQL и не делай ставку на legacy RPC. В Odoo 19 есть CLI/shell для работы с ORM, а внешний старый RPC уже считается устаревающим направлением; для новой внешней интеграции Odoo продвигает JSON-2 API, но для вашей миграции это всё равно хуже, чем внутренний ORM. ([Odoo][3])

Главный принцип:
**Bitrix-данные нормализуются в staging-слой → затем грузятся в стандартные Odoo-модели через ORM → все исходные идентификаторы Bitrix сохраняются в `x_bitrix_*` полях и/или mapping-таблицах.**

### 3) Как понимать источник Bitrix

#### 3.1 Проекты и рабочие группы

Источник — `b_sonet_group`.
Поле `PROJECT='Y'` отделяет проекты от рабочих групп, а `CLOSED='Y'/'N'` — фактически отражает закрыт/не закрыт. У источника **нет нормальной сущности “стадия проекта”**; `P`-стадии в `b_tasks_stages` — это внутренние системные периоды для Ганта, а не бизнес-статус проекта. У проектных тегов отдельная логика: `b_sonet_group_tag` хранит только `GROUP_ID + NAME`, без собственного ID. 

#### 3.2 Задачи

Источник — `b_tasks`, с привязкой к проекту через `GROUP_ID`, родителю через `PARENT_ID`, стадии через `STAGE_ID`, дедлайну через `DEADLINE`, описанию через `DESCRIPTION`. Исполнители собираются из `b_tasks_member`, где `TYPE='R'` — ответственный, `TYPE='A'` — соисполнитель; в вашей выгрузке берутся оба типа, и подзадачи уже включены. Важно: `STAGE_ID` часто пустой, поэтому в Odoo значительная часть задач должна быть создана без стадии, а не с выдуманной дефолтной стадией. 

#### 3.3 Теги

Теги задач в Bitrix **не глобальные**: одна и та же метка дублируется разными строками `b_tasks_label`. Поэтому маппинг должен делаться **по имени**, а не по `TAG_ID`. В README у вас прямо зафиксировано, что дедупликация идет по `NAME`, и уникальных имён 214. Отдельно есть проектные теги из `b_sonet_group_tag`; в Odoo это должен быть один объединённый справочник тегов. 

#### 3.4 Стадии

`b_tasks_stages` — это **не этапы проектов**, а канбан-стадии задач. Типы там разные: `G` — стадии групп, `U` — личные, `P` — системные периоды проекта, `A` — потоки. Для первой волны миграции используй только `G`-стадии, потому что именно они логически совпадают с задачными стадиями Odoo и в README уже показано связывание `s.ENTITY_TYPE = 'G' AND s.ENTITY_ID = g.ID`. 

#### 3.5 Комментарии

Комментарии задач берутся из `b_forum_message` через `t.FORUM_TOPIC_ID = fm.TOPIC_ID`, причём критичен фильтр `FORUM_ID = 11` для производительности. Переносить как историю общения нужно **только реальные комментарии**, то есть записи с `SERVICE_TYPE IS NULL` и `NEW_TOPIC = 'N'`; автосообщения и системные логи не нужно тащить в chatter как обычные комментарии. В README также есть отдельный источник комментариев встреч через `b_sonet_log_comment` + `b_sonet_log` с `EVENT_ID='calendar'`. 

#### 3.6 Встречи

Встречи берутся из `b_calendar_event`, но в Bitrix для каждого участника существует копия события. Поэтому дедупликация обязательна: брать записи только с `ID = PARENT_ID`, а список участников собирать из дочерних строк, где `PARENT_ID = ce.ID AND ID != PARENT_ID`. Организатор — `MEETING_HOST`. 

#### 3.7 Вложения

Вложения не лежат “напрямую” в задачах и комментариях: они проходят через модуль Disk, то есть `b_disk_attached_object → b_disk_object → b_file`. Для задач, комментариев и встреч используются разные `ENTITY_TYPE`, а `file_path` — это **относительный** путь вида `/upload/...`; для реальной миграции нужен файловый снапшот Bitrix-сервера.

### 4) Целевая модель Odoo

Мигрируй в такие модели Odoo:

* Bitrix project/workgroup → `project.project`
* Bitrix task/subtask → `project.task`
* Bitrix task stage → `project.task.type`
* Bitrix tags → `project.tags`
* Bitrix task comments → chatter через `mail.thread` / `mail.message`
* Bitrix files → `ir.attachment`
* Bitrix meetings → `calendar.event`

Это соответствует нативной архитектуре Odoo: стадии задач используются как колонки Kanban и по умолчанию проект-специфичны; подзадачи в Odoo поддерживаются стандартно; chatter и вложения реализованы стандартными messaging-механизмами Odoo. ([Odoo][2])

### 5) Правила маппинга Bitrix → Odoo

#### 5.1 Проекты / рабочие группы

Каждую запись из `b_sonet_group` создавай как `project.project`. Не пытайся разделять на разные модели.
Сохраняй кастомные поля:

* `x_bitrix_id`
* `x_bitrix_type` (`project` / `workgroup`)
* `x_bitrix_closed`
* `x_bitrix_owner_bitrix_id`

Поле “статус проекта” не маппить в отдельную project-stage модель, потому что в источнике её нет. Если проект закрыт (`CLOSED='Y'`), отражай это через технический флаг, архивирование или отдельное кастомное поле, но не выдумывай fake stage. Основание: в исходнике единственный надёжный признак завершённости — `CLOSED`, а `P`-стадии не являются бизнес-стадиями проекта. 

#### 5.2 Теги

Создай единый справочник тегов Odoo.
Дедуплицируй строго по `name`.
Не используй `TAG_ID` Bitrix как основной ключ, потому что одна и та же метка повторяется разными ID. Проектные теги из `b_sonet_group_tag` включай в тот же справочник, помечая источник через `x_bitrix_tag_source`, если это нужно для аудита. 

#### 5.3 Стадии задач

Мигрируй в `project.task.type` только `G`-стадии.
Стадии создавай **с привязкой к проектам**, а не как глобальные по умолчанию, потому что и в Bitrix, и в Odoo логика стадий в первую очередь проектная. Если одинаковые стадии реально используются множеством проектов и это подтверждено бизнесом, допускается объединение, но по умолчанию храни их проект-специфично. Odoo прямо документирует, что task stages по умолчанию привязаны к проекту, хотя могут быть shared.  ([Odoo][2])

#### 5.4 Задачи

Загружай в `project.task`:

* `name`
* `project_id`
* `description`
* `date_deadline`
* `stage_id`
* `parent_id`
* assignees
* tags
* кастомные `x_bitrix_*`

Подзадачи создавай не в первом проходе, а во втором:
сначала создаются все задачи без `parent_id`, потом вторым проходом выставляются связи родитель/дочерняя. В Odoo подзадачи являются стандартной функцией и должны быть реализованы именно так, а не в виде самодельной иерархии в текстовом поле.  ([Odoo][1])

#### 5.5 Исполнители

Не хардкодь внутренние имена полей модели Odoo “по памяти”; сначала определи их через `env['project.task']._fields`. Но логика должна быть такой: Bitrix `R` + `A` маппятся в Odoo assignees, а автор постановки (`O`) и наблюдатели (`U`) не маппятся автоматически в assignees. В Odoo задачи поддерживают multiple assignees.  ([Odoo][4])

#### 5.6 Комментарии

Реальные комментарии задач грузить в chatter задачи.
Системные сообщения и автосообщения не переносить как обычный пользовательский chatter. Максимум — сохранить их в отдельную архивную модель или JSON-лог, если бизнес настаивает. Это критично, иначе chatter в Odoo превратится в мусор. Основание: в источнике чётко различаются реальные комментарии, автосообщения и системные логи. 

#### 5.7 Вложения

Вложения задач крепи к задаче.
Вложения комментариев — либо к `mail.message`, либо, если технически это окажется нестабильно при массовой загрузке, к задаче с сохранением `x_bitrix_forum_message_id` и ссылкой на исходный комментарий.
Вложения встреч — к `calendar.event`.
Файлы брать только из реального файлового снапшота сервера Bitrix, потому что SQL даёт лишь относительный путь. 

#### 5.8 Встречи

Создавай `calendar.event` только из дедуплицированного набора `ID=PARENT_ID`, а участников добавляй отдельным этапом по собранному списку `OWNER_ID` дочерних записей. Комментарии встреч загружай в chatter события отдельной фазой, если они действительно нужны бизнесу.

### 7) Staging-слой

Сделай промежуточный нормализованный слой между Bitrix и Odoo. Не грузить “сырой SQL-результат сразу в ORM”.

Для каждой сущности создай Python DTO / Pydantic-модель:

* `BitrixProject`
* `BitrixTask`
* `BitrixStage`
* `BitrixTag`
* `BitrixComment`
* `BitrixAttachment`
* `BitrixMeeting`

На этом слое:

* чистятся NULL/пустые строки;
* нормализуются даты;
* удаляются дубли;
* строятся связи;
* отбрасываются недопустимые записи;
* готовится audit-лог.

Для дат соблюдай форматы Odoo ORM для Date/Datetime полей. Odoo отдельно предупреждает, что с date/datetime надо обращаться аккуратно и использовать корректный серверный формат. ([Odoo][5])

### 8) Порядок миграции и зависимости

Выполняй строго в таком порядке:

1. миграция пользователей/контактов и таблицы соответствия Bitrix user → Odoo user/partner;
2. миграция объединённого справочника тегов;
3. миграция проектов;
4. миграция стадий задач;
5. миграция задач без родительских связей;
6. второй проход: parent/subtask links;
7. миграция комментариев задач;
8. миграция вложений задач;
9. миграция вложений комментариев;
10. миграция встреч;
11. миграция комментариев встреч;
12. миграция вложений встреч;
13. reconciliation и отчёт по расхождениям.

Такой порядок обязателен, потому что в Odoo связанные записи должны ссылаться на уже существующие сущности; это согласуется и с логикой импортов/external id в Odoo, и с самой последовательной моделью загрузки данных. ([Odoo][6])

### 9) Идемпотентность и mapping tables

Обязательное требование: миграция должна быть **идемпотентной**.

Нужно создать mapping-таблицы либо хранить `x_bitrix_*` ключи так, чтобы повторный прогон не создавал дублей. Минимально:

* Bitrix project/group ID → Odoo project ID
* Bitrix task ID → Odoo task ID
* Bitrix stage ID → Odoo stage ID
* Bitrix tag normalized name → Odoo tag ID
* Bitrix forum message ID → Odoo mail.message ID
* Bitrix calendar event ID → Odoo calendar.event ID
* Bitrix file checksum/path → Odoo attachment ID
* Bitrix user ID → Odoo user ID / partner ID

External IDs в Odoo служат именно для устойчивой идентификации записей независимо от database ID; эту же идею используй для внутренних mapping-таблиц мигратора. ([Odoo][7])

### 10) Стратегия загрузки через ORM

Используй batch processing и ручные коммиты.

Рекомендованные батчи:

* projects/tags/stages: 100–500
* tasks: 500–2000
* comments: 1000–5000
* attachments: 100–500
* meetings: 200–1000

После каждого батча:

* `env.cr.commit()`
* логируй количество созданных/обновлённых/пропущенных записей
* сохраняй checkpoint
* при ошибке откатывай только текущий батч

Odoo shell даёт доступ к `env`, курсору и ORM, поэтому этот паттерн естественен для серверной миграции. ([Odoo][3])

### 11) Правила по комментариям и chatter

Так как chatter в Odoo строится на `mail.thread`, не создавай “фальшивые текстовые поля истории” внутри задачи. История должна быть либо в стандартном chatter, либо в отдельной технической архивной модели, если речь о системных логах. Задачи и события, для которых нужен chatter, должны оставаться в стандартной парадигме `mail.thread`. ([Odoo][8])

Нужно реализовать два режима:

* `preserve_authorship=True` — попытка назначить автора исходного комментария через найденный Odoo user/partner;
* `fallback_system_author=True` — если автора нет в mapping, использовать системного миграционного пользователя, но сохранить `x_bitrix_author_id`.

### 12) Что не копировать 1:1 из Bitrix

Не пытайся воссоздать в Odoo:

* личные стадии пользователя (`U`);
* системные периоды проекта (`P`);
* потоковые стадии (`A`) в первой версии;
* технические уведомления Bitrix как пользовательские комментарии;
* прямую структуру Disk-модуля Bitrix;
* дубли календарных событий по каждому участнику.

В Odoo это должно быть преобразовано в нормальную модель Odoo, а не перенесено как есть. Основание: архитектурно эти сущности в Bitrix либо внутренние, либо служебные, либо не имеют прямого 1:1 соответствия в Project/Calendar Odoo.

### 13) Тестовый план

Сделай 3 уровня проверки.

**Уровень 1. Dry run**

* ничего не записывает в Odoo;
* считает объёмы;
* проверяет обязательные связи;
* выводит missing mappings.

**Уровень 2. Pilot migration**

* 3–5 проектов;
* 500–1000 задач;
* 2000–5000 комментариев;
* 100–200 вложений;
* 20–50 встреч.

**Уровень 3. Full migration rehearsal**

* полный прогон в staging Odoo;
* сравнение counts и выборочная ручная верификация.

Обязательные проверки после загрузки:

* число проектов;
* число задач;
* число подзадач;
* число задач без проекта;
* число комментариев;
* число встреч;
* число вложений;
* число orphan records;
* число пользователей без маппинга;
* число задач с потерянной стадией;
* число комментариев без автора;
* число файлов, которых нет на диске.

---

### 14) Acceptance criteria

Считать миграцию успешной, если:

* все проекты/группы перенесены в `project.project`;
* не менее 99.5% задач связаны с проектом корректно;
* parent/subtask связи восстановлены;
* реальные комментарии задач отображаются в chatter;
* системные сообщения не засоряют chatter;
* вложения открываются из Odoo;
* встречи не дублируются;
* повторный запуск не создаёт дублей;
* есть финальный reconciliation-report.

## Мои уточняющие рекомендации к этому плану

Я бы для первой производственной волны поставил такой scope:
**проекты + теги + стадии G + задачи + подзадачи + реальные комментарии задач + вложения задач/комментариев**.
Встречи и комментарии встреч вынес бы либо во вторую волну, либо запускал отдельным модулем после стабилизации проектного блока. У вас календарная часть в Bitrix архитектурно заметно более “шумная”, чем задачи. 

И ещё одно важное решение: на этапе проектирования пусть Claude **не угадывает названия Odoo-полей по памяти**, а проверяет их через ORM-интроспекцию (`env['model']._fields`, `fields_get()`, test create/write на staging). Это особенно важно для assignees, chatter-связей и calendar attendees, где в разных версиях и конфигурациях легко ошибиться. Это уже не вопрос бизнес-логики, а вопрос надёжности реализации.
